## 기술 회고
- Q: ApiResponse 래핑 방식에서 성공/실패를 어떻게 구분했나요?
- A: `success` 필드로 구분하고, 실패 시 `error` 객체에 `code`, `message`, `details`를 담았다. 클라이언트가 `success` 필드 하나만 보고 분기할 수 있도록 설계했다.

- Q: GlobalExceptionHandler에서 어떤 예외를 어떻게 처리했나요?
- A: `BusinessException`은 `ErrorCode` enum 기반으로 HTTP 상태코드와 메시지를 매핑했고, `MethodArgumentNotValidException`은 필드별 에러를 `details` 배열로 내려주도록 처리했다. 예상치 못한 예외는 `COMMON-007`로 500을 반환한다.

- Q: PATCH와 PUT 중 PATCH를 선택한 이유는?
- A: 수정 시 변경할 필드만 보내는 부분 수정 방식이 실무에 더 적합하다. PUT은 전체 필드를 보내야 하므로 불필요한 데이터 전송이 발생한다.
