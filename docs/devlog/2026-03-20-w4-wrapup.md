## 기술 결정

- 비관적 락 선택 (TASK-016): 티켓팅은 동시에 수십 명이 하나의 좌석을 선점하려는 구조라 충돌이 잦음. 낙관적 락은 재시도 로직이
  필요해 복잡도가 높아지므로 DB 레벨에서 선점하는 비관적 락을 먼저 적용
- Redis 분산락으로 전환 (TASK-019): 비관적 락은 DB 커넥션을 점유해 트래픽 폭증 시 DB 부하가 집중됨. Redis 분산락은
  DB 외부에서 락을 관리하므로 부하 분산 가능하고 멀티 인스턴스 환경에서도 동작
- 락 키 설계 `hold:seat:{showtimeId}:{seatId}`: 같은 회차의 같은 좌석에 대한 요청만 직렬화하고 다른 좌석 요청은
  서로 영향을 주지 않도록 설계
- Redisson 선택: RLock이 tryLock/unlock 인터페이스를 제공해 구현이 명확하고 Lettuce 기반 직접 구현보다 안정적
- LAZY 로딩 명시: @ManyToOne 기본값이 EAGER라 불필요한 연관 엔티티가 항상 로드되고 있었음. LAZY로 전환 후 필요한
  경우만 fetch join으로 명시적 로드

## 기술 회고

- Q: 비관적 락에서 Redis 분산락으로 전환한 이유는?
- A: 비관적 락은 SELECT FOR UPDATE로 DB 커넥션을 점유한다. 티켓팅처럼 트래픽이 몰리는 환경에서는 커넥션 풀이 고갈될 수
  있다. Redis 분산락은 DB 외부에서 락을 관리하므로 DB 부하를 분산할 수 있고 멀티 인스턴스 환경에서도 동작한다.

- Q: leaseTime을 5초로 설정한 이유는?
- A: 서버 장애 시 락을 획득한 프로세스가 unlock을 호출하지 못해 락이 영구적으로 걸리는 것을 방지하기 위해 자동 해제 시간을
  설정했다. 비즈니스 로직 수행 시간보다 충분히 긴 값으로 설정했다.

- Q: N+1 문제가 발생한 원인은?
- A: HoldExpirationService에서 만료 대상 Hold 목록을 조회한 후 루프에서 getShowtimeSeat()를 호출할 때
  LAZY 로딩으로 인해 Hold마다 추가 쿼리가 발생했다. fetch join으로 한 번에 로드해 쿼리를 N+1번에서 1번으로 줄였다.

- Q: 동시성 테스트를 어떻게 구성했나요?
- A: ExecutorService + CountDownLatch로 10개 스레드를 동시에 출발시켜 동일 좌석에 선점 요청을 보냈다. 성공
  카운트가 1개인지, DB에서 HELD 상태인 showtime_seat이 1개인지 이중으로 검증했다.
