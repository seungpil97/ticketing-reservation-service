## 기술 결정

- Redis 캐시 적용 대상 선택: 이벤트 목록은 자주 조회되고 변경이 적으며 모든 사용자에게 동일한 데이터를 반환해 캐시 효과가 가장 큼.
  좌석 조회는 상태가 수시로 바뀌어 캐시 미적용
- JWT Stateless 인증: 티켓팅은 분산 환경이 전제되므로 세션 방식보다 서버 상태를 저장하지 않는 JWT가 수평 확장에 유리
- 블랙리스트 TTL = AccessToken 잔여 만료 시간: 이미 만료된 토큰 키를 Redis에 계속 보관하면 메모리 낭비. 잔여 시간만큼만
  유지하면 자동 정리됨
- RefreshToken Rotation: 재발급마다 Redis 토큰이 교체되어 기존 토큰이 즉시 무효화됨. 탈취 감지 시
  deleteRefreshToken() 선호출 후 예외 throw로 해커 보유 토큰까지 차단
- 에러코드 인터페이스 분리: 단일 enum에 모든 도메인 에러가 혼재하면 도메인 경계가 불명확하고 이름 충돌 위험이 증가함. ErrorCode
  인터페이스로 공통 계약을 추출하고 도메인별 enum이 구현하도록 설계

## 기술 회고

- Q: JWT를 선택한 이유는? 세션 방식과 비교하면?
- A: 세션은 서버에 상태를 저장하므로 인스턴스가 늘어날수록 세션 동기화 문제가 생긴다. JWT는 서버가 상태를 저장하지 않아 어떤 인스턴스가
  요청을 받아도 동일하게 검증할 수 있다. 다만 JWT는 발급 후 만료 전 강제 무효화가 어렵다는 단점이 있어 Redis 블랙리스트로
  보완했다.

- Q: AccessToken 만료 시간을 15분으로 설정한 이유는?
- A: 탈취 시 피해 범위를 최소화하기 위해 짧게 설정했다. RefreshToken으로 재발급하는 구조이므로 사용자 경험에 큰 영향 없이
  보안을 강화할 수 있다.

- Q: Rotation에서 탈취 감지 시 deleteRefreshToken()을 예외 throw 전에 호출하는 이유는?
- A: 예외가 먼저 throw되면 deleteRefreshToken()이 실행되지 않는다. Redis 삭제를 먼저 처리해 해커가 보유한 새
  RefreshToken도 무효화한 후 예외를 발생시켜야 강제 로그아웃이 완전히 보장된다.

- Q: 에러코드를 인터페이스 기반으로 설계한 이유는?
- A: 단일 enum은 도메인이 늘어날수록 파일이 비대해지고 다른 도메인 개발자가 에러코드를 추가할 때 충돌 위험이 있다. 인터페이스로 공통
  계약(getStatus, getCode, getMessage)만 정의하고 각 도메인 enum이 독립적으로 구현하면 확장성과 도메인 경계가
  명확해진다.
