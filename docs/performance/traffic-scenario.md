# 트래픽 시나리오 및 목표 TPS 정의

## 1. 작성 목적

이 문서는 부하 테스트(TASK-037)와 동시성 제어 설계(TASK-016, TASK-019)의
수치 기반 근거를 제공하기 위해 작성되었다.

"얼마나 버텨야 하는가"에 대한 기준이 없으면 부하 테스트 결과는 해석 불가능하며,
비관적 락 vs Redis 분산락 선택, 대기열 도입 등의 기술 결정도 설명하기 어렵다.

이 문서의 수치는 실제 티켓팅 서비스(인터파크, 멜론티켓 등)의 공개된 트래픽 패턴과
미드레인지 서비스의 일반적인 특성을 참고하여 설정한 현실적 가정이다.

## 3. 목표 TPS 정의

| 구간       | 목표 TPS  | 동시 사용자 수 | 근거                                |
|----------|---------|----------|-----------------------------------|
| 정상 트래픽   | 50 TPS  | 500명     | 일반 공연 예매 산발적 요청, 사용자당 평균 응답 대기 포함 |
| 티켓 오픈 폭증 | 500 TPS | 20,000명  | 오픈 직후 10초 내 집중 요청, 대기열 진입 포함      |

**목표 응답시간**

| 구간       | p95 응답시간   | p99 응답시간   |
|----------|------------|------------|
| 정상 트래픽   | 200ms 이하   | 500ms 이하   |
| 티켓 오픈 폭증 | 1,000ms 이하 | 3,000ms 이하 |

> **응답시간 목표 수치 근거**
> - p95 200ms: Google UX 연구 기준, 사용자가 "즉각적인 응답"으로 인지하는 임계값
> - p99 500ms: Google Core Web Vitals 기준, 정상 트래픽에서 허용 가능한 응답 상한선
> - p95 1,000ms: Nielsen Norman Group 기준, 사용자가 이탈 없이 기다릴 수 있는 한계
> - p99 3,000ms: 티켓 오픈 특성상 선점 동기가 강해 평소보다 대기 허용치가 높음을 반영한 폭증 구간 상한선. Google 조사에서
    3초 초과 시 이탈률 급등 데이터 참고

> 폭증 구간에서 p99 3초 기준은 "사용자가 버튼을 누른 뒤 응답 없음으로 이탈하기 전" 임계값으로 설정했다.

## 4. 요청 분포

티켓 오픈 서비스의 특성상 조회 요청이 예약/결제보다 압도적으로 많다.
오픈 직전 좌석 조회가 집중되고, 오픈 직후 선점 요청이 폭증하는 패턴을 반영했다.

| API                       | 정상 트래픽 비율 | 오픈 폭증 비율 | 설명                                                                                 |
|---------------------------|-----------|----------|------------------------------------------------------------------------------------|
| GET /events               | 20%       | 5%       | 이벤트 목록 조회 (Redis 캐시 적용). 진입 후 상세로 이동하므로 반복 호출 낮음. 폭증 구간엔 이미 목적 이벤트를 알고 접속하므로 비율 급감 |
| GET /showtimes/{id}/seats | 40%       | 50%      | 좌석 현황 확인. 예매 과정에서 새로고침 반복 패턴으로 가장 높은 비율. 폭증 구간엔 오픈 직전 상태 확인이 집중되어 비율 더 높아짐         |
| POST /queue/enter         | 0%        | 20%      | 대기열 진입. 정상 구간엔 대기열 불필요. 오픈 직후 20,000명이 동시에 진입 요청을 보내는 구간                           |
| POST /showtimes/{id}/hold | 15%       | 20%      | 좌석 선점 (Redis 분산락 경합 발생 구간). 조회 대비 전환율 낮음. 폭증 구간엔 대기열 통과 후 선점 시도가 집중                |
| POST /holds/{id}/reserve  | 15%       | 3%       | 예약 확정. 정상 구간엔 선점:확정 비율 약 1:1 가정. 폭증 구간엔 선점까지만 도달하고 확정은 이후 구간으로 분산                  |
| POST /payments            | 10%       | 2%       | 결제 처리. 예약 후 이탈률 약 30% 가정 (가격 재확인, 망설임 등). 폭증 구간엔 확정과 동일하게 이후 구간으로 분산               |

### 요청 분포 상세 근거

**정상 트래픽**

| API                       | 비율  | 근거                                                |
|---------------------------|-----|---------------------------------------------------|
| GET /events               | 20% | 서비스 진입 시 이벤트 목록부터 보는 패턴. 한 번 조회 후 상세로 이동하므로 비율 낮음 |
| GET /showtimes/{id}/seats | 40% | 좌석 현황을 반복 확인하는 패턴 (새로고침). 예매 과정에서 가장 많이 호출됨       |
| POST /showtimes/{id}/hold | 15% | 조회 후 실제 선점으로 이어지는 전환율. 조회 대비 낮은 게 자연스러움           |
| POST /holds/{id}/reserve  | 15% | 선점한 사람 대부분이 예약 확정까지 진행한다고 가정 (hold:reserve ≈ 1:1) |
| POST /payments            | 10% | 예약 후 결제 이탈률 약 30% 가정 (망설임, 가격 확인 등)               |

**티켓 오픈 폭증**

| API                       | 비율  | 근거                                      |
|---------------------------|-----|-----------------------------------------|
| GET /events               | 5%  | 오픈 직후엔 이미 목적지(이벤트)를 알고 들어옴. 목록 조회 거의 없음 |
| GET /showtimes/{id}/seats | 50% | 오픈 직전 좌석 상태 확인이 폭발적으로 집중. 새로고침 반복 패턴    |
| POST /queue/enter         | 20% | 대기열 진입 요청이 오픈 직후 한꺼번에 몰림                |
| POST /showtimes/{id}/hold | 20% | 대기열 통과 후 선점 시도 집중                       |
| POST /holds/{id}/reserve  | 3%  | 폭증 구간에서는 선점까지만 도달하는 경우가 대부분. 확정은 이후 구간  |
| POST /payments            | 2%  | 동일한 이유                                  |

## 5. 시나리오 정의

### 시나리오 A: 정상 트래픽

- 대상: 일반적인 공연 예매 시간대 (오픈 후 1시간 이후)
- 동시 사용자: 500명
- 요청 패턴: 산발적, 균등 분포
- 검증 목표: 정상 구간에서 p95 200ms 이하 유지

```
k6 설정 기준:
  vus: 50
  duration: 5m
  ramp-up: 30s
```

### 시나리오 B: 티켓 오픈 폭증

- 대상: 티켓 오픈 직후 0~60초
- 동시 사용자: 최대 20,000명 (점진적 증가 후 급감)
- 요청 패턴: 좌석 조회 → 대기열 진입 → 선점 집중
- 검증 목표: 500 TPS에서 에러율 1% 이하, 좌석 중복 선점 0건

```
k6 설정 기준:
  stages:
    - duration: 10s, target: 200   // 급등
    - duration: 30s, target: 500   // 폭증 유지
    - duration: 20s, target: 50    // 급감
```

### 시나리오 C: 대기열 포함 전체 플로우

- 대상: 오픈 ~ 예약 확정 ~ 결제 전체 플로우
- 흐름: 로그인 → 대기열 진입 → 순번 확인 → 선점 → 예약 확정 → 결제
- 검증 목표: 전체 플로우 성공률 99% 이상, 좌석 정합성 보장

## 6. 기술 선택과 목표 수치의 연결

### 비관적 락 → Redis 분산락 전환 이유

비관적 락(DB 레벨 `SELECT ... FOR UPDATE`)은 단일 서버에서는 동작하지만
500 TPS 폭증 구간에서 DB 커넥션 풀 고갈과 락 대기 스레드 누적 문제가 발생한다.

- DB 커넥션 풀 기본값: HikariCP 10개
- 500 TPS 중 선점 요청(20%) = 100 TPS → 각 요청이 DB 락을 점유하면 큐잉 발생
- Redis 분산락은 DB 커넥션 외부에서 락을 관리하므로 DB 부하를 분리할 수 있음
- Redisson tryLock + TTL로 무한 대기 없이 빠른 실패(fail-fast) 처리 가능

### 대기열 도입 이유

티켓 오픈 폭증 시나리오에서 20,000명이 동시에 선점 API를 호출하면
Redis 분산락 경합이 극단적으로 증가하고 HOLD TTL 내에 처리 불가능한 요청이 쌓인다.

- 대기열 없이 500 TPS 선점 요청을 직접 처리: 락 경합 → 타임아웃 → 에러율 급등
- Sorted Set 기반 대기열로 요청을 순번화 → 선점 처리량을 서버가 감당 가능한 수준으로 조절
- 사용자에게 "현재 N번째 대기 중" 피드백 제공 → 이탈 방지
